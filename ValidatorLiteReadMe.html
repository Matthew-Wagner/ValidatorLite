Welcome to ValidatorLite

ValidatorLite is a light weight, single file HTML 5 / javascript library for adding validation to HTML 5 forms.

While jQuery and Bootstrap are not required, this sample file does make use of them.

The library itself uses the HTML 5 Constraint Validation API but expands on it with some new attributes.

To use ValidatorLite, include the script in your HTML file, add the novalidation attribute to your form and initialize ValidatorLite on your form element. 

<form id="MyForm" name="MyForm" novalidation>
    <input id="TextBox1" name="TextBox1" type="text" placeholder="TextBox1" />
    <button id="SubmitButton" name="SubmitButton">Submit</button>
</form>
<script type="text/javascript">
    (const | var | let) myNewValidator = new ValidatorLite("MyForm");
</script>

The standard behavior for ValidatorLite is to create a new label with the class of “feedback” for the fields being validated, then it will append the label to the field. ValidatorLite will then add onblur events to each input and select element on the form and an onsubmit handler for the form itself. The onblur handler calls the default checkValidity() method of the element and adds the “is-valid” or “is-invalid” class to the element itself and all labels linked to the element using the for attribute. The onsubmit handler does the same thing but also prevents the form from being submitted if any of the fields are invalid.

ValidatorLite will add two new methods to each element found by calling form.querySelectAll with the fieldselector configuration (defaults to “input,select,textarea”). The two new methods are:

  • reportValidityLite() Updates the UI for the selected element
  • checkValidityLite(report = false) Return true or false after combining the default checkValidity() with any custom validation functionality. If report is true, reportValidityLite() will also be called on the element

These are the standard Constraint Validation API attributes that are supported by ValidatorLite:

  • novalidation This is set on the form itself or individual elements, this needs to be set in order to use the custom validation in ValidatorLite
  • required: Ensures the field is not empty.
  • pattern: Uses a Regular Expression (RegEx) to validate the input format (e.g., pattern="[0-9]{3}" for three digits).
  • type: Automatically validates common formats like email, url, and number.
  • min / max / step: Restricts the range and increments of numeric or date inputs. (Currently not supported)
  • minlength / maxlength: Controls the number of characters allowed. (Currently not supported)

These are custom data- attributes used by ValidatorLite

  • data-vl-no-report Add this attribute (does not require a value) to suspend updating the UI for this element. Defaults to false in the configuration for all fields
  • data-vl-add-feedback Set to any value and a new label element will be created and added to the DOM. Defaults to true in the configuration for all fields
  • data-vl-feedback-id The id of an element that will be used to display the relevant message. This also prevents ValidatorLite from creating a new label. The element pointed to here will be used for all messaging and styling
  • data-vl-keep-text The text in the corresponding feedback element will not be altered. Should be used with data-vl-feedback-id, otherwise the default text of the newly created label (blank) will always be kept. Use this if you have a static error message you want to use for every case. Defaults to false in the configuration for every field
  • data-vl-append-text The corresponding validation message will be appended, with a leading space, to the initial text of the feedback element. Again, use this with data-vl-feedback-id to display a dynamic message that begins with the initial text of the feedback element
  • data-vl-custom-validator The name of a function to be called when validating. ValidatorLite wraps the function call in a promise and expects a return of either true or false. Use async and await if you need to make any asynchronous calls inside the function
  • data-vl-remote-url The remote URL that will be called for validation. ValidatorLite uses the fetch promise to check for a success response. If there is an error or the response status is not “OK”, the field is marked as invalid and the remote message is displayed. If you need to make a remote call and then manipulate the data that is returned, use data-vl-custom-validator with the async and await keywords. The loading class will be added to the feedback element and the loading message will be displayed while awaiting the fetch
  • data-vl-remote-method the method to be used for the data-vl-remote-url. Defaults to “GET”
  • data-vl-remote-fields a comma seperated list of field ids that will make up the data for the remote fetch. If the data-vl-remote-method is set to “POST”, the fields will be included as JSON objects in the data parameter. Otherwise the field names and values will be converted to name = value pairs and included in the URL

The default messages for each validation can be modified in the configuration or on the elements themselves. The default messages and their data- attributes are as follows:

  • data-vl-custom-message Custom validation returned false
  • data-vl-required-message This field is required
  • data-vl-pattern-message The field did not match the expected pattern
  • data-vl-type-message The field did not match the expected type
  • data-vl-success-message Empty by default
  • data-vl-remote-message The remote call failed
  • data-vl-loading-message Loading...

This is the default configuration for ValidatorLite: 

{
    onblur: true,
    onsubmit: true,
    report: true,
    feedbackplacement: "after",
    fieldselector: "input,select,textarea",
    css: {
        error: "is-invalid",
        success: "is-valid",
        loading: "loading",
        feedback: "feedback"
    },
    messages: {
        custom: "Custom validation returned false",
        required: "This field is required",
        pattern: "The field did not match the expected pattern",
        type: "The field did not match the expected type",
        success: "",
        remote: "The remote call failed",
        loading: "Loading...",
        minlength: "The minimum length of the field has not been met"
    }
}

  • onblur: Used with report, adds an “onblur” event listener to the field, which triggers validation when the field loses focus.
  • onsubmit: Adds an “onsubmit” event listener to the form, which triggers validation when the form is submitted. This will also prevent the form from being submitted if any fields are invalid.
  • report Determines whether to update the UI by default. Setting this to false and leaving onblur true could cause some unusual side-effects
  • feedbackplacement Expects either before or after, which determines where the feedback element will be placed relative to the field.
  • fieldselector The CSS selector that will be used on the form to retrieve all the elements for validation
